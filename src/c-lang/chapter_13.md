# chapter 13 複数のファイルによる開発

## 13.1 開発効率の壁

プログラムが大きくなると、1 つのファイルに全部を書き続けるのはつらくなります。  
そこで、役割ごとにファイルを分けます。C の実用開発ではごく自然な流れです。

## 13.2 C言語のビルドシステム

C のソースコードは、そのまま実行されるわけではありません。  
一般に、コンパイルとリンクを通して実行ファイルになります。

- コンパイル: 各ソースファイルを機械語に近い形へ変換する
- リンク: 複数の部品をつなぎ、最終的な実行ファイルを作る

`gcc` や `clang` で次のようにビルドできます。

```bash
gcc -std=c23 -Wall -Wextra -O2 main.c util.c -o app
```

`-std=c23` は C23 を使う指定、`-Wall -Wextra` は警告を増やす指定です。  
初学者ほど警告を大切にすると、ミスに早く気づけます。

## 13.3 プリプロセッサ

`#include` や `#define` は、コンパイルの前段で動く仕組みです。  
これをプリプロセッサと呼びます。

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

マクロは便利ですが、書き方を誤ると予期しない動作を招きます。  
関数で書けるなら関数、どうしても必要なときだけマクロ、と考えると安全です。

## 13.4 マクロ処理

マクロは単純な置換に見えて、実際にはかなり強力です。  
ただし、強力さは危険と表裏一体です。式の評価順や副作用には注意しましょう。

```c
#define SQUARE(x) ((x) * (x))
```

`SQUARE(i++)` のような使い方は危険です。  
`x` が何度も評価されるからです。初学者のうちは、マクロより関数を優先するほうがわかりやすいことが多いです。

## 13.5 ソースコード分岐処理

`#ifdef` などを使うと、条件に応じてコードを切り替えられます。

```c
#ifdef DEBUG
    printf("debug mode\n");
#endif
```

これは、デバッグ用の表示や、環境ごとの切り替えで使われます。  
ただし、コードが増えすぎると読みづらくなるので、使いどころを選びましょう。

## 13.6 コンパイラとリンカの仕事

コンパイラは各ファイルの中身を理解し、リンカはそれらをつなぎます。  
ヘッダファイルには宣言を書き、`*.c` ファイルには定義を書きます。

```c
/* util.h */
#ifndef UTIL_H
#define UTIL_H

int add(int a, int b);

#endif
```

```c
/* util.c */
#include "util.h"

int add(int a, int b) {
    return a + b;
}
```

```c
/* main.c */
#include <stdio.h>
#include "util.h"

int main(void) {
    printf("%d\n", add(2, 3));
    return 0;
}
```

## 13.7 ライブラリの利用

標準ライブラリだけでなく、外部ライブラリを使うとできることが広がります。  
ただし、初学者のうちは「標準ライブラリをしっかり使える」ことが先です。標準の道具を理解してから外部ライブラリへ進むと、学びがつながります。

## 13.8 第13章のまとめ

- 大きなプログラムでは複数ファイルに分けます
- コンパイルとリンクの役割を知ることが大切です
- `#include` と `#define` はプリプロセッサの仕組みです
- ヘッダには宣言、`*.c` には定義を書きます

## 13.9 練習問題

1. `add` 関数を `util.c` と `util.h` に分けてみましょう。  
2. `#define` で定数を作ってみましょう。  
3. `-Wall -Wextra` をつけてコンパイルしてみましょう。
